# F1. Rocket Management

## Problem definition

### Context

The administrative staff needs a tool to register and maintain the inventory of AstroBookings' spacecraft fleet. Without proper management of the rockets and their individual characteristics (seat capacity and operational range), it is impossible to plan launches and control reservation volumes.

### User stories

- As a `user` I want to **register new rockets with their seat capacity and operational range** so that I can _have them available when scheduling future launches_
- As a `user` I want to **view the list of all available rockets** so that I can _manage the inventory of the current fleet_
- As a `user` I want to **modify the details of a rocket** so that I can _correct errors or update its capacity and range_
- As a `user` I want to **deactivate a rocket** so that I can _remove it from the active fleet without losing its history_

---

## Solution sketch

A simple UI and API to consult and introduce the vehicles in the fleet, keeping track of the name, capacity, and range of each rocket.

### Model

| Entity | Attribute | Type | Constraints |
|--------|----------|-------------|-------------|
| `Rocket` | `id` | `uuid#` | Autogenerated, unique |
| `Rocket` | `name` | `string#` | min 3, max 100 characters |
| `Rocket` | `capacity` | `integer` | min 1, max 10 |
| `Rocket` | `range` | `enum` | Values: Earth, Moon, Mars |

> legend: `#` for unique, `?` for optional

### Back

- MUST handle incoming requests, validate payload and persist rockets (creation and update).
- MUST return a JSON list of all active rockets in the system.
- MUST validate that the range is one of the allowed enum values.
- MUST implement soft deletion for deactivating rockets.
- MUST NOT allow the creation or modification of rockets resulting in a duplicate name.

### Front

- MUST show a specific rocket fleet management screen with a summary list/table.
- MUST provide a form for creating new rockets, applying validations before submitting data.
- MUST allow access and management to any user without role or operative restrictions in this phase.
- MAY optionally show success messages or alerts after creation.

### Contract

| Method | Endpoint | Success | Error |
|--------|----------|---------|-------|
| `GET` | `/api/rockets` | `200` | `500` |
| `POST` | `/api/rockets` | `201` | `400, 409` |
| `PUT` | `/api/rockets/:id` | `200` | `400, 404, 409` |
| `DELETE` | `/api/rockets/:id` | `204` | `404` |

**Request — `CreateRocket` / `UpdateRocket`**

```json
{
  "name": "string",
  "capacity": "number",
  "range": "Earth | Moon | Mars"
}
```

**Response — `Rocket`**

```json
{
  "id":      "uuid",
  "name": "string",
  "capacity": "number",
  "range": "Earth | Moon | Mars"
}
```

**Error response**

```json
{
  "code":    "400",
  "error":   "validation_failed",
  "message": "Capacity must be greater than 0"
}
```

---

## Acceptance criteria

- [ ] the system SHALL require name, seat capacity, and a valid range (Earth, Moon, Mars) for every rocket
- [ ] WHEN a user submits valid data for a rocket the system SHALL register the vehicle, assign a unique id, and make it available in the system
- [ ] WHEN a user modifies an existing rocket the system SHALL update its data ensuring name uniqueness is maintained
- [ ] WHEN a user deactivates a rocket the system SHALL mark it as inactive (soft delete) without deleting the physical record
- [ ] IF a user attempts to assign an invalid range THEN the system SHALL reject the operation with a validation error
- [ ] IF a user attempts to assign an invalid capacity THEN the system SHALL reject the operation with a validation error

---

## Out of scope

- Physical deletion of records from the database (historical data must be kept forever).
- Assignment of rockets to launches (this belongs to the specific flight feature).
